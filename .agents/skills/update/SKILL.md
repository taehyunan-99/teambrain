# update — 가이드를 현재 코드에 맞춰 인터뷰로 채워간다

## 입력 형태

| 호출 | 동작 |
|------|------|
| `/update` (Codex는 `$update`) | 전 영역 일반 갱신. 9개 인터뷰 + 6개 충돌 검증 |
| `/update <영역경로>` | 특정 영역만 갱신 (예: `/update apps/api`) |
| `/update --restructure` | 영역 경계 재구성 전용 (별도 흐름). `references/restructure-flow.md` 절차 적용 |
| `/update --restructure <영역>` | 특정 영역의 재구성만 진행 |

## 언제 실행하는가

- 베이스라인 완성 직후 (가이드 본격 채우기 시작)
- 주요 기능 추가 후 (새 패턴이 컨벤션이 될지 결정)
- 버그 fix 클러스터 발생 후 (반복 실수를 LEARNED_CAUTIONS에 박제)
- 외부 의존성 교체 후 (HOW/COMMANDS 갱신)
- 온보딩·릴리즈 직전 (가이드 최신성 보장)

피해야 할 시점:
- init 직후 (코드가 비어있어 추출할 사실 없음)
- 대규모 리팩터링 진행 중 (완료 후 실행)
- PR 머지 직전 (인터뷰 동반, 시간 소요)

## 모드 분기 (절차 진입 직전)

호출 인자에 `--restructure` 가 있으면 **별도 흐름**으로 진입한다. 일반 update 절차(아래 1)~6))는 실행하지 않는다.

- `--restructure` 흐름은 `references/restructure-flow.md` 의 10단계 절차를 그대로 따른다.
- 영역 경계 재구성은 비가역에 가까우므로 한 번에 한 영역만 처리한다.
- LEARNED_CAUTIONS.md 항목 이전·외부 강결합 `@import` 재연결까지 모두 사용자 확인을 거친다.
- 완료 후 일반 `/update` 를 다시 실행하면 새로 생성된 영역들의 HOW/HOW NOT/WHY를 인터뷰로 채울 수 있다.

`--restructure` 가 없으면 아래 일반 update 절차로 진행한다.

## 절차 (일반 update)

### 1) 사전 검증

- 현재 디렉토리가 git repo인지 확인 (아니면 경고하되 계속 진행)
- root map(`CLAUDE.md` 또는 `AGENTS.md`)이 존재하는지 확인. 없으면 `/agentic-project-init`을 먼저 실행하라고 안내하고 중단.
- 직전 update 이후의 변화 추출:
  - `git log --since=<직전 update> --name-status` 로 변경 파일 목록
  - 직전 update가 없으면 init 이후 전체 변경분
- 미해소 충돌이 남아있는지 확인 (`.update-pending-conflicts.md` 또는 직전 보고서). 있으면 그것부터 재질문.

### 2) 영역 목록 확보

- root map의 "영역별 가이드" 섹션을 읽어 영역 목록 추출
- 각 영역의 본문 가이드 + `LEARNED_CAUTIONS.md` 경로 확인
- 영역마다 변경 감지 수행 (1단계의 변경 파일을 영역별로 분류)

### 3) 영역별 인터뷰 (질문 유형 카탈로그 적용)

각 영역에 대해 `references/question-types.md`의 9개 유형을 다음 순서로 검사·인터뷰:

1. **변경 사실 확인** (자동 반영 + 큰 변화는 제안)
2. **삭제·이전 확인** (확인 후 제거)
3. **외부 의존성 변화** (제안)
4. **컨벤션 합의** (제안)
5. **안티패턴 예측** (HOW NOT 추가 제안)
6. **암묵지 추출** (WHY 추가 제안)
7. **드리프트 감지** (HOW 충돌 해소, 변경 이력 주석 필수)
8. **신규 LEARNED CAUTIONS 후보** (질문형, 사용자 메모 입력 후에만 추가)
9. **영역 경계 재검토** (의견 제시만, 일반 update에서는 결정 X)

**진행 원칙**:
- 한 영역의 모든 유형 질문을 **묶어서** 제시 (매 항목마다 묻지 않는다)
- 각 제안에 **근거 한 줄 명시** (커밋 ID, 파일 경로, 발견 횟수 등)
- "지금 안 정함" 선택지 항상 포함 → `<!-- TBD -->` 마커
- 한 영역 인터뷰 완료 시 그 영역 가이드를 **즉시 저장** (중단 시 손실 방지)

**섹션별 갱신 정책은 `references/section-policy.md` 참조.**
- AI 갱신: CONTENTS, WHERE(작은 변화), COMMANDS(빌드/테스트/린트)
- 제안: WHAT, HOW, WHERE(큰 변화)
- 보존: HOW NOT, WHY, LEARNED_CAUTIONS.md (추가만 제안, 자동 수정 절대 금지)

### 4) 교차 정합성 검증 (Deep 해소)

모든 영역 인터뷰가 끝난 후 `references/conflict-types.md`의 6개 충돌 유형을 순서대로 검사:

1. 컨벤션 모순
2. WHERE 중복/누락
3. HOW NOT vs HOW 모순
4. 용어 불일치
5. 의존 방향 충돌
6. COMMANDS 충돌

**검출 시**:
- 사용자에게 충돌 목록을 일괄 보고
- 각 충돌마다 **Deep 인터뷰** (단순 "어느 쪽?"이 아닌 "의도된 차이인가? 의도라면 양쪽에 명시")
- 해소된 충돌은 양쪽 가이드 갱신 (변경 이력 주석 포함)

**미해소 시**:
- update는 **미완료**로 표시
- 미해소 충돌 목록을 저장 (다음 /update에서 가장 먼저 재질문)
- 마무리 안내에서 미완료 사실을 명시

### 5) 자동 반영 리포트

마무리 직전, "AI 갱신" 모드로 자동 반영된 항목을 사용자에게 리포트:

```
자동 반영된 변경:
  CONTENTS:
    + apps/api/users.py 추가
    - apps/api/legacy.py 제거
  COMMANDS:
    ~ 테스트 명령 갱신 (pytest → uv run pytest)
  WHERE:
    + apps/api → libs/shared 의존 추가

이상 발견 시 `git diff` 로 확인 가능합니다.
```

### 6) 마무리 안내

```
update 결과:
  ✅ 영역 인터뷰 완료 (N건 반영)
  {✅ 교차 정합성 검증 통과 | ⚠️ 교차 정합성 검증: 충돌 M건 미해소 → update 미완료}

자동 반영: K건 (위 리포트 참조)
사용자 결정 반영: L건

다음 단계:
  - git diff 로 변경 확인
  - 사용자가 직접 커밋 (자동 커밋 안 함)
  - 미해소 충돌은 다음 /update 실행 시 가장 먼저 재질문됩니다.
```

## 사용자 결정 보호 원칙 (절대)

- **사용자 결정 사항 자동 덮어쓰기 금지** — 한 번 사용자가 답한 내용은 자동으로 바꾸지 않는다.
- **구조/설계 변경으로 덮어써야 하면 반드시 사용자 확인** — "이 결정을 변경해야 합니다. 진행할까요?" 명시 질문.
- **변경 이력 주석 보존** — 결정이 바뀌면 이전 결정을 HTML 주석으로 남긴다.
  ```markdown
  <!-- prev: snake_case (2025-11 결정) → camelCase (2026-05 결정) -->
  ```
- **`LEARNED_CAUTIONS.md`는 절대 자동 수정·삭제 금지** — 사용자/learn 전용 자산. update는 신규 후보(질문 유형 9)만 사용자 메모 입력 후 추가.

## 영역 경계 재구성 (별도 명령 안내)

일반 `/update`에서 6번 유형(영역 경계 재검토)이 검출되면 **안내만** 출력하고 결정·실행 안 함:

```
영역 경계 재검토 권장:
  - src/utils/ 가 87개 파일을 넘고 책임이 섞여 있어 보입니다.

별도 명령 `/update --restructure src/utils` 로 진행하시기를 권장합니다.
일반 update에서는 가이드 변경 없이 다음 유형으로 넘어갑니다.
```

`--restructure` 흐름은 `references/restructure-flow.md` 의 10단계 절차를 따른다. 영역 분할/병합/이름 변경/삭제 + 파일 이동 + 가이드 재배치 + LEARNED_CAUTIONS 항목 이전 + 외부 강결합 `@import` 재연결까지 모두 사용자 확인을 거치며, 한 번에 한 영역만 처리한다.

## 금지

- **명시 호출 없이 자동 발동 금지** — 사용자가 `/update` (또는 `$update`)을 직접 입력하지 않은 상황에서 선제 발동 금지.
- **사용자 결정 사항 자동 덮어쓰기 금지** — 위 보호 원칙 참조.
- **`LEARNED_CAUTIONS.md` 자동 수정·삭제 금지** — 신규 후보 추가도 사용자 메모 입력 후에만.
- **자동 커밋 금지** — 사용자가 직접 검토 후 커밋.
- **충돌 미해소 상태로 update 완료 처리 금지** — 미해소 1건이라도 있으면 "미완료" 표시.
- **본문 가이드의 사용자 작성분을 휴리스틱만으로 갱신 금지** — 모호하면 사용자에게 묻는다.

- `references/conflict-types.md` — 6개 교차 정합성 충돌 유형 + Deep 해소 절차
