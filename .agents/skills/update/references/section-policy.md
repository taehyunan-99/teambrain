# 섹션별 갱신 정책

8섹션 + `LEARNED_CAUTIONS.md` 별도 파일에 대한 update 시 행동 규칙.

## 분류 체계

세 가지 모드:

- **AI 갱신**: 사실 기반. 변경 감지 시 자동 반영. 리포트 끝에 "이 항목들은 자동 반영됨" 목록으로 사용자가 확인 가능.
- **제안**: AI가 초안/변경안 제시. 사용자 확인 후에만 반영. 이전 결정이 있으면 변경 이력 주석으로 남김.
- **보존**: 사용자 결정 자산. 자동 수정 절대 금지. 추가만 제안 (질문형). 변경 필요 시 반드시 사용자 확인.

## 절대 원칙

- **사용자 결정 사항 자동 덮어쓰기 금지** — 한 번 사용자가 답한 내용은 자동으로 바꾸지 않는다.
- **구조/설계 변경으로 덮어써야 할 때도 반드시 사용자 확인** — "이 사용자 결정을 바꿔야 합니다. 진행할까요?" 명시적 질문.
- **변경 이력 주석 보존** — 사용자 결정이 바뀌면 이전 결정을 HTML 주석으로 남긴다 (예: `<!-- prev: X (2025-11) → Y (2026-05) -->`).

| **CONTENTS** | AI 갱신 | 파일·모듈 목록은 사실. 자동 반영 + 리포트에 표시. 큰 변화(영역 분할 가능성)는 6번 질문 유형으로 전환. |
| **HOW** | 제안 | 컨벤션. 드리프트 감지(질문 유형 5) 시 사용자 확인 후 갱신. 변경 이력 주석 필수. |
| **HOW NOT** | 보존 (+ 제안 추가만) | 자동 수정 금지. 안티패턴 예측(질문 유형 3)으로 추가만 제안. 기존 항목은 사용자 명시 동의 없이 변경/삭제 X. |
| **WHERE** | AI 갱신 (큰 변화는 제안) | 파일 이동/삭제는 자동 반영. 강결합 `@import` 추가·제거는 사용자 확인. |
| **WHY** | 보존 (+ 제안 추가만) | 의도·배경은 코드에서 추론 불가. 암묵지 추출(질문 유형 4)로 추가만. 기존 내용 자동 수정 X. |
| **COMMANDS** | AI 갱신 (가드는 보존) | package.json/Makefile 등에서 추출되는 빌드/테스트/린트는 자동 반영. 사용자가 추가한 영역 고유 가드는 보존. |

## "AI 갱신" 시 정확한 행동

대상 섹션: CONTENTS, WHERE(작은 변화), COMMANDS(빌드·테스트·린트)

1. 변경 감지 (직전 update 이후 추가/삭제/변경된 사실)
2. **사용자 결정 사항이 함께 변경되는지 확인** — 같은 줄에 사용자가 직접 쓴 설명/주석이 있으면 그 줄은 자동 갱신 대상 제외
3. 변경 반영
4. update 마무리 리포트에 "자동 반영됨" 목록으로 표시:
   ```
   자동 반영된 변경:
     - CONTENTS: src/api/users.py 추가
     - CONTENTS: src/api/legacy.py 제거
     - COMMANDS: 테스트 명령 갱신 (pytest → uv run pytest)
   ```
5. 사용자가 리포트 보고 이상 발견 시 되돌릴 수 있도록 git diff 안내

## "제안" 시 정확한 행동

대상 섹션: WHAT, HOW, WHERE(큰 변화)

1. 변경 후보 생성 (코드 스캔 결과 + 근거)
2. 인터뷰로 제시 (`AskUserQuestion` 객관식 + "Other" + "지금 안 정함")
3. 사용자 응답에 따라:
   - **새 결정**: 해당 줄을 가이드에 작성. 변경 이력 주석은 없음.
   - **기존 결정 변경**: 이전 줄을 HTML 주석으로 보존 + 새 줄 작성.
     ```markdown
     <!-- prev: snake_case (2025-11 결정) → camelCase (2026-05 결정) -->
     - 명명 규칙: camelCase
     ```
   - **지금 안 정함**: `<!-- TBD -->` 마커 + 다음 update에서 재질문

## "보존" 시 정확한 행동

대상 섹션: HOW NOT, WHY, LEARNED_CAUTIONS.md

1. **기존 내용 절대 자동 수정·삭제 금지**
2. 신규 후보만 인터뷰로 제시 (질문 유형 3, 4, 9)
3. 사용자가 명시적으로 "이 항목 삭제/수정해주세요" 라고 요청한 경우에만 변경
4. 변경 시에도 이전 항목을 변경 이력 주석으로 보존:
   ```markdown
   <!-- prev (2025-11~2026-05): "fetch 직접 호출 금지" — 2026-05 ApiClient 도입으로 폐기 -->
   ```

## 사용자 작성분 식별 방법

AI가 어디까지 갱신해도 되는지 판단하려면 "사용자가 직접 쓴 부분"을 식별해야 한다.

### Heuristic

- **HTML 주석 마커 사용 (권장)**: 사용자 추가분에 `<!-- user-added: YYYY-MM-DD -->` 또는 `<!-- locked -->` 마커. update는 이 마커가 있는 블록을 자동 갱신 대상에서 제외.
- **`learn` 스킬 누적 형식**: `- (YYYY-MM-DD) <메모>` 형식은 LEARNED_CAUTIONS.md에만 있으므로 식별 쉬움.
- **placeholder 잔존 여부**: `_(update 스킬에서 채워질 자리)_` 가 남아 있으면 사용자가 아직 작성 안 함 → 새 내용 작성 가능.
- **변경 이력 주석**: `<!-- prev: ... -->` 가 있으면 사용자 결정이 한 번 이상 있었음 → 신중.

### 모호하면 사용자에게 묻는다

위 휴리스틱으로 판단이 모호하면 자동 갱신하지 않는다. 인터뷰에서 "이 내용을 갱신해도 될까요?" 라고 명시적으로 묻는다.

## LEARNED_CAUTIONS.md / HOW NOT 분리 제안 (임계치 기반)

`LEARNED_CAUTIONS.md`는 이미 별도 파일이지만 HOW NOT은 본문 가이드에 머문다.
HOW NOT이 길어지면 update 인터뷰에서 분리를 제안한다.

**임계치**: HOW NOT 섹션이 30줄(또는 본문 가이드 100줄 도달)을 넘으면 제안.

**제안 형식**:
> apps/backend의 HOW NOT이 42줄로 길어졌습니다. 별도 파일로 분리할까요?
> - 분리 (`@./how-not.md` import 구조로 전환) / 유지 / 지금 안 정함

분리 결정 시:
- 새 파일 `apps/backend/how-not.md` 생성
- 본문 가이드의 HOW NOT 섹션은 `@./how-not.md` 한 줄 + 마크다운 링크 fallback (LEARNED_CAUTIONS와 동일 하이브리드)
- 기존 항목 전체 이동, 변경 이력 주석 보존

WHY도 동일 임계치 규칙 적용 가능 (사용자가 원하면).
