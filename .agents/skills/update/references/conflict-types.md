# 교차 정합성 충돌 유형 카탈로그

update의 모든 영역 인터뷰가 끝난 후, 영역 간 가이드가 서로 모순되지 않는지 검증한다.
6개 충돌 유형 모두 검사. Deep 해소 방식으로 인터뷰. 미해소 시 update는 **미완료**로 표시.

## 검사 대상

- 모든 영역의 본문 가이드 (AGENTS.md / CLAUDE.md)
- root map 파일
- `LEARNED_CAUTIONS.md`는 검사 대상에서 제외 (사용자 누적 자산, 자동 수정 금지)

## Deep 해소란

단순 "어느 쪽이 맞나요?" 가 아니라 **충돌 원인까지 파고들어 사용자에게 묻는다**:

- 의도된 차이인가, 실수인가?
- 의도된 차이라면 양쪽 가이드에 **명시적으로** 기록 (예: "이 영역에선 X, 저 영역에선 Y, 이유는 Z")
- 실수라면 한쪽을 정답으로 정하고 다른 쪽 갱신

---

## 1. 컨벤션 모순

**탐지**: 두 영역의 HOW가 같은 주제에 대해 다른 규칙을 명시.

**예시**:
- A 영역 HOW: "오류는 raise" / B 영역 HOW: "오류는 Result 반환"
- 두 영역이 서로 호출하거나 같은 도메인 모델을 공유한다면 충돌.

**Deep 인터뷰**:
> A·B 영역의 오류 처리 컨벤션이 다릅니다.
> - A: raise (HTTP boundary) / B: Result 반환 (도메인 로직)
> 이 차이는 의도된 건가요?
> - 의도됨 → 양쪽 HOW에 "boundary는 raise, 도메인은 Result, 변환은 X 어댑터에서" 라고 명시
> - 실수 → 어느 쪽으로 통일할지 결정
> - 지금 안 정함 (충돌 미해소)

---

## 2. WHERE 중복/누락

**탐지**:
- 같은 파일이 두 영역의 WHERE에 강결합(`@import`)으로 등장
- 또는 어느 영역의 WHERE에도 안 잡힌 핵심 파일이 존재

**예시**:
- `shared/schema.py`가 `apps/frontend/`와 `apps/backend/` 양쪽 WHERE에서 `@import`
- 또는 root map에 명시된 `infra/` 폴더가 어느 영역에도 매핑 안 됨

**Deep 인터뷰**:
> `shared/schema.py`가 frontend·backend 양쪽 WHERE에서 강결합 import 됩니다.
> SoT는 어디인가요?
> - frontend가 SoT / backend가 SoT / 양쪽이 동등 (별도 영역으로 분리 권장)
> - 지금 안 정함

---

## 3. HOW NOT vs HOW 모순

**탐지**: 한 영역의 HOW에서 권장하는 패턴이 다른 영역의 HOW NOT에서 금지됨.

**예시**:
- A 영역 HOW: "외부 API 호출 시 fetch 직접 사용"
- B 영역 HOW NOT: "fetch 직접 호출 금지 — 항상 ApiClient 래퍼 사용"

**Deep 인터뷰**:
> A 영역은 fetch 직접 호출을 권장하는데, B 영역은 같은 패턴을 금지합니다.
> 영역마다 정책이 다른 게 의도된 건가요?
> - 의도됨 → 양쪽에 "이 영역에서만 적용" 명시
> - 실수 → 어느 쪽이 정답인지 결정
> - 지금 안 정함

---

## 4. 용어 불일치

**탐지**: 같은 개념을 영역마다 다른 이름으로 부름. WHAT/HOW/WHERE에서 등장하는 핵심 명사 비교.

**예시**:
- A 영역: "Request body" / B 영역: "Payload" / C 영역: "Input DTO"
- 동일한 데이터 구조를 가리키는데 영역마다 명칭이 다름

**탐지 방법**:
- 1차: 단순 텍스트 매칭 (같은 객체가 다른 이름으로 등장하는지)
- 2차: 사용자가 의심 사례 제시 시 확인 (자동 임베딩 분석은 비용 큼, MVP에서는 제외)

**Deep 인터뷰**:
> 같은 데이터 구조를 영역마다 다르게 부르는 것 같습니다.
> - A: Request body / B: Payload / C: Input DTO
> 통일 용어가 있나요?
> - "Payload"로 통일 / "Request body"로 통일 / "Input DTO"로 통일 / 의도된 차이 / 지금 안 정함

---

## 5. 의존 방향 충돌

**탐지**:
- A 영역의 WHERE/HOW가 B 영역에 의존한다고 적혀있는데, B 영역의 가이드는 A를 모름.
- 또는 양방향 의존 (A → B → A 순환).

**예시**:
- frontend WHERE: "backend의 /api/users 사용"
- backend WHERE: 언급 없음 (frontend가 자신을 호출한다는 정보 없음)
- 또는 frontend → backend → frontend (계층 위반 가능성)

**Deep 인터뷰**:
> frontend는 backend에 의존한다고 명시되어 있지만, backend 가이드는 frontend를 모릅니다.
> backend의 "피의존" 항목에 frontend를 추가할까요?
> - 추가 / backend는 클라이언트를 모르는 게 맞음 (이유 입력 → WHY에 기록) / 지금 안 정함

순환 의존이 감지되면:
> A → B → A 순환 의존이 감지됐습니다. 계층 위반 가능성이 있습니다.
> - 의도된 양방향 (이유 입력) / 한 방향 끊어야 함 / 지금 안 정함

---

## 6. COMMANDS 충돌

**탐지**: 같은 작업(테스트/빌드/린트)에 영역마다 다른 명령이 안내됨.

**예시**:
- A 영역 COMMANDS: "테스트: pytest -v"
- B 영역 COMMANDS: "테스트: python -m unittest"
- root map 공통 명령어 가드와도 다를 수 있음

**Deep 인터뷰**:
> 테스트 명령이 영역마다 다릅니다.
> - A: pytest -v / B: python -m unittest / root 공통: 없음
> 영역별로 정말 다른가요, 통일해야 하나요?
> - 영역별로 다름 (각 영역 도구 차이) / 통일 (어느 명령으로) / root 공통으로 이동 / 지금 안 정함

---

## 검증 절차

1. **모든 영역 인터뷰 완료 후 실행** — 인터뷰 도중 검증하면 변경분이 미반영
2. **6개 유형 순서대로 검사** — 1 → 2 → 3 → 4 → 5 → 6
3. **충돌 발견 시 사용자에게 일괄 보고**:
   ```
   교차 정합성 검사: 충돌 4건 발견
     [1] 컨벤션 모순: apps/frontend HOW vs apps/backend HOW (오류 처리)
     [2] WHERE 중복: shared/schema.py SoT 모호
     [3] 용어 불일치: Payload vs Request body
     [6] COMMANDS 충돌: 테스트 명령 영역별 상이

   각 충돌에 대해 Deep 인터뷰를 진행합니다.
   ```
4. **충돌별 Deep 인터뷰**
5. **해소된 충돌은 양쪽 가이드 갱신** — "사용자 결정 사항" 규칙 적용 (자동 덮어쓰기 금지 원칙은 새 결정에는 해당 없음, 단 이전 결정이 있으면 변경 이력 주석으로 남김)
6. **미해소 충돌이 1건이라도 있으면**:
   - update는 **미완료**로 표시
   - 미해소 충돌 목록을 `.update-pending-conflicts.md` 또는 콘솔에 남김
   - 다음 update 실행 시 가장 먼저 재질문

## 미해소 시 표시 방법

update 마무리 안내:
```
update 결과:
  ✅ 영역 인터뷰 완료 (12건 반영)
  ⚠️ 교차 정합성 검증: 충돌 2건 미해소 → update 미완료

미해소 충돌은 다음 /update 실행 시 가장 먼저 재질문됩니다.
```

미완료 상태는 가이드 파일에 별도 마커를 박지 않는다 — 다음 update에서 다시 검출되므로 충분.
