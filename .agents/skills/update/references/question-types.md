# update 인터뷰 질문 유형 카탈로그

9개 유형. 각 유형은 (탐지 트리거 / 질문 형식 / 영향 섹션 / 근거 명시 방법)을 따른다.
영역별로 일괄 묶어 제시한다 — 매 항목마다 묻지 않는다.

## 공통 규칙

- **근거 명시 필수**: 모든 제안 끝에 "이 제안을 하는 이유"를 한 줄로 붙인다 (예: "최근 PR #12 / commit a3f2c1에서 이 패턴이 3번 등장").
- **선택지 형식**: 가능하면 `AskUserQuestion` 객관식으로 제시. "지금 안 정함" 선택지를 항상 포함 — 가이드에 `<!-- TBD -->` 마커로 남기고 다음 update에서 재질문.
- **자유 입력 허용**: 객관식이 부족하면 "Other"로 자유 입력 받기.

---

## 1. 변경 사실 확인 (AI 갱신 + 큰 변화 제안)

**트리거**: 파일/폴더/엔드포인트/모듈이 직전 update 이후 추가·삭제됨.

**질문 형식**:
> `src/api/`에 엔드포인트 3개 추가됐어요 (POST /users, GET /users/:id, DELETE /users/:id). CONTENTS·WHERE 갱신할까요?
> - 모두 반영 / 일부만 반영 / 지금 안 정함

**영향 섹션**: CONTENTS, WHERE, COMMANDS

**근거 예시**: "직전 update 후 신규 파일 5개 추가됨"

**행동 규칙**:
- 단순 추가/삭제는 자동 반영하되 update 끝 리포트에 "이 항목들은 자동 반영됨" 목록으로 보여준다.
- 영역 경계가 바뀔 만한 변화(파일 50개+ 추가, 새 서브폴더 도입)는 자동 반영 금지 → 6번 유형으로 전환.

---

## 2. 컨벤션 합의 (제안)

**트리거**: 동일 목적에 2개 이상의 구현 패턴이 공존.

**질문 형식**:
> 오류 처리에 try/except와 Result 패턴이 섞여 있어요.
> - try/except 통일 / Result 통일 / 영역별로 다르게 유지 / 지금 안 정함

**영향 섹션**: HOW

**근거 예시**: "controllers/에서 try/except 5건, services/에서 Result 3건 검출 — 같은 영역 안에서 혼재"

**행동 규칙**:
- 어느 쪽이 우세한지 정량 정보 제시 (X건 vs Y건).
- 사용자가 한쪽 선택하면 HOW에 한 줄로 명시 + 반대 패턴은 HOW NOT에 "이 영역에서는 금지" 항목으로 추가할지 별도 질문.

---

## 3. 안티패턴 예측 (제안 → HOW NOT 추가)

**트리거**: 일반적으로 위험한 패턴이 코드에 출현 (직접 DB 호출, any 회피, 동기 IO in async 핸들러 등).

**질문 형식**:
> DB 호출이 컨트롤러에 직접 들어가는 패턴이 보여요 (`src/api/user_controller.py:42`).
> HOW NOT에 "컨트롤러에서 직접 DB 호출 금지" 추가할까요?
> - 추가 / 의도된 패턴이라 추가 안 함 / 지금 안 정함

**영향 섹션**: HOW NOT

**근거 예시**: "controller 레이어에서 SQL 직접 호출 3건 (file_a:12, file_b:55, file_c:88) — 일반적으로 서비스 레이어 위반"

**행동 규칙**:
- "왜 안 되는지" 한 줄을 항상 함께 적도록 유도. 이유 없는 룰은 LLM이 흘려보낸다.
- 사용자가 "의도된 패턴"이라 답하면 WHY에 그 이유를 기록할지 후속 질문.

---

## 4. 암묵지 추출 (제안 → WHY 추가)

**트리거**: 특이한 선택이 일관되게 보임 (한 영역만 비동기 미사용, 특정 라이브러리만 회피 등).

**질문 형식**:
> 이 영역(`analysis/`)만 asyncio를 안 쓰는데, 의도된 건가요?
> - 의도됨 (이유 입력) / 의도 안 됨 (개선 필요) / 모름 / 지금 안 정함

**영향 섹션**: WHY

**근거 예시**: "다른 영역 4개 중 3개가 async를 쓰는데 analysis만 sync only"

**행동 규칙**:
- 의도된 선택이면 WHY에 한 줄로 기록.
- 모르겠다고 답하면 `<!-- TBD -->` 마커 + 다음 update에서 재질문.

---

## 5. 드리프트 감지 (제안 → HOW 충돌 해소)

**트리거**: 기존 가이드의 HOW 와 실제 코드 패턴이 불일치.

**질문 형식**:
> HOW에는 "snake_case 사용"인데 최근 추가된 파일 절반이 camelCase예요 (`utils/parseDate.ts` 등 8건).
> 어느 쪽이 정답인가요?
> - HOW를 camelCase로 갱신 / 코드를 snake_case로 정리 / 영역별 분리 / 지금 안 정함

**영향 섹션**: HOW (사용자 결정 사항 갱신에 해당하므로 **사용자 확인 후에만** 변경)

**근거 예시**: "HOW 갱신일: 6개월 전 / 그 후 추가된 파일 12개 중 8개가 다른 규칙"

**행동 규칙**:
- 사용자가 HOW 변경 결정 시: 이전 HOW는 변경 이력 주석으로 남김 (`<!-- prev: snake_case (2025-11 결정) → camelCase (2026-05 결정) -->`).
- "코드를 정리"는 update 범위 밖. 후속 작업으로 안내만.

---

## 6. 영역 경계 재검토 (의견 제시만)

**트리거**: 한 영역의 파일 수가 임계치(예: 50+) 초과 또는 서로 다른 책임이 한 폴더에 섞임.

**질문 형식 (의견 제시만, 결정 X)**:
> `src/utils/`가 87개 파일을 넘고 책임이 섞여 있어요 (날짜·문자열·HTTP·DB 헬퍼 혼재).
> 영역 분할이 필요해 보입니다. `/agentic-project-init update --restructure` 로 별도 진행을 권장합니다.

**영향 섹션**: 없음 (일반 update에서는 가이드 변경 안 함)

**근거 예시**: "파일 수 87개, 책임 카테고리 4개 추정"

**행동 규칙**:
- 일반 update에서 6번은 **안내만** 출력하고 즉시 다음 유형으로 넘어간다.
- 사용자가 그 자리에서 restructure를 원하면 별도 명령 실행을 안내.

---

## 7. 삭제·이전 확인 (AI 갱신 + 확인)

**트리거**: 가이드에 명시된 파일/패턴이 코드에서 사라짐.

**질문 형식**:
> WHERE에 적힌 `auth/legacy.py`가 삭제됐어요. 가이드에서 제거할까요?
> - 제거 / 이전 위치 정보를 추가 / 지금 안 정함

**영향 섹션**: CONTENTS, WHERE

**근거 예시**: "WHERE에 명시된 파일 1개 / 현재 git tree에 없음 / 마지막 commit a3f2: 'remove legacy auth'"

**행동 규칙**:
- 자동 제거가 아니라 확인 후 제거. 사용자가 작성한 의존성 설명 줄이 함께 사라지지 않도록 주의.

---

## 8. 외부 의존성 변화 (제안)

**트리거**: `package.json` / `requirements.txt` / `pyproject.toml` / `go.mod` 등의 변경 감지.

**질문 형식**:
> axios → fetch로 교체됐어요. HOW의 HTTP 호출 예시와 COMMANDS의 mock 명령어가 영향받습니다.
> 갱신할까요?
> - 모두 갱신 / 일부 / 지금 안 정함

**영향 섹션**: HOW, COMMANDS

**근거 예시**: "package.json에서 axios 제거 + fetch 사용 처 12건"

**행동 규칙**:
- 의존성 교체는 HOW에 영향이 크므로 사용자 확인 후 갱신.
- COMMANDS의 빌드/테스트 명령이 바뀌었으면 자동 반영하되 리포트에 알림.

---

## 9. 신규 LEARNED CAUTIONS 후보 (질문형 톤, 추가만 제안)

**트리거**: 직전 update 이후 한 영역에 `fix` / `bug` / `hotfix` / `revert` 커밋 다수 (예: 5건+).

**질문 형식 (단정하지 않고 질문형)**:
> 지난 주 `src/api/`에 fix 커밋 5건이 있었어요 (#a3f, #b21, #c44, #d09, #e17). 공통 원인이 있나요?
> 있다면 LEARNED_CAUTIONS.md에 한 줄로 박제하는 게 좋습니다.
> - 공통 원인이 있어요 (메모 입력) / 각자 다른 원인 / 지금 안 정함

**영향 섹션**: LEARNED_CAUTIONS.md (제안만, 사용자가 메모 입력해야 추가)

**근거 예시**: "직전 update 후 7일간 fix 류 커밋 5건 발생"

**행동 규칙**:
- AI가 단정해서 작성하지 않는다 ("이런 실수가 반복돼 보입니다" 금지).
- 사용자가 메모를 입력하면 `learn` 스킬의 누적 형식(`- (YYYY-MM-DD) <메모>`)으로 LEARNED_CAUTIONS.md에 추가.
- 사용자가 "공통 원인 없음"이라 답하면 가이드 변경 없음.

---

## 인터뷰 진행 순서

1. **영역별 묶기**: 한 영역에 대한 질문을 모두 묶어서 한 번에 제시. "src/api/ 영역에 5개 제안이 있어요"
2. **유형 순서**: 1(변경 사실) → 7(삭제) → 8(의존성) → 2(컨벤션) → 3(안티패턴) → 4(암묵지) → 5(드리프트) → 9(LEARNED 후보) → 6(영역 경계, 안내만)
3. **건너뛰기**: 사용자가 "이 유형 전체 스킵" 요청 가능. 다음 update에서 재시도.
4. **저장 빈도**: 각 영역 인터뷰 완료 후 그 영역 가이드를 즉시 저장 (중단 시 손실 방지).
