# 장애 회고: 중복 결제 발생 (INC-204)

- 발생: 2026-04-21 21:30 ~ 22:10 (40분)
- 영향: 결제 1,203건 중 87건 중복 청구
- 작성: 이준호

## 무슨 일이 있었나
프로모션 트래픽 급증으로 Redis 응답 지연. 멱등성 키 조회(`idem:{key}`)가 타임아웃나자, 코드가 "키 없음 = 신규 요청"으로 처리해 그대로 PG 승인을 진행했다. 같은 키의 재시도가 전부 신규 결제로 통과.

## 근본 원인
- 멱등성 체크가 **Redis 단일 의존**. Redis 장애 = 멱등성 전체 붕괴.
- 타임아웃 시 fail-open(통과) 설계. 결제에선 fail-closed(거부)가 맞다.

## 조치
- 즉시: Redis 타임아웃 시 fail-closed로 변경(결제 거부 후 클라 재시도 유도). 핫픽스 배포 완료.
- 영구: 멱등성 키를 Redis 캐시 + **DB 유니크 제약 2중화**로. Redis 미스여도 DB INSERT가 중복을 막는다. → 3/11 결정(Redis 단독)을 뒤집는 후속 결정 필요. (link: 2026-04-29 결정)

## 교훈
- 결제 경로의 외부 의존은 fail-closed가 기본.
- 멱등성은 캐시가 아니라 **영속 저장소의 유니크 제약**이 최종 방어선이어야 한다.
