# 결정: 멱등성 키 — 클라이언트 생성 방식 채택

- 일시: 2026-03-11
- 결정자: 박서연(제안), 김도현(승인)

## 배경
결제 승인 API에 같은 요청이 두 번 들어오면(네트워크 재시도, 더블클릭) 중복 결제가 발생한다. 멱등성 키로 막아야 한다.

## 검토한 대안
1. **클라이언트 생성 (채택)**: 클라이언트가 UUID를 `Idempotency-Key` 헤더로 보냄. 서버는 키+요청바디 해시를 24h 저장, 중복이면 캐시된 응답 반환.
2. 서버 생성: 주문ID 기반으로 서버가 키 생성. → 클라 재시도를 구분 못 함(같은 주문의 정당한 재결제도 막힘). 탈락.
3. 자연 키(주문ID+금액): 부분 환불·재결제 시 충돌. 탈락.

## 결정
- **클라이언트가 `Idempotency-Key` 헤더로 UUID 전송.**
- 서버는 Redis에 `idem:{key}` → 응답 스냅샷, TTL 24h.
- 키 없는 결제 요청은 400 거부.

## 근거
- 클라 재시도와 정당한 재결제를 클라가 구분해 키를 줄 수 있음.
- Stripe 등 업계 표준 방식.

## 리스크
- 클라이언트가 키를 매번 새로 만들면 멱등성 무력화 → SDK에서 키 생성 강제하기로(action: 이준호).
