# 장애 회고: 정산 이중 송금 (INC-231)

- 발생: 2026-05-12 04:00 배치
- 영향: 가맹점 3곳에 정산금 이중 송금(총 1,840만원), 당일 회수
- 작성: 이준호

## 무슨 일이 있었나
정산 배치가 04:00에 시작했으나 송금 API 지연으로 04:50까지 안 끝남. 크론 모니터가 "배치 멈춤"으로 오판해 수동 재실행. 1차 배치가 아직 송금 중이었고, 2차가 같은 정산일로 다시 송금 → 일부 가맹점 이중 송금.

## 근본 원인
- 정산 멱등성(가맹점ID+정산일)은 **명세서 생성 단계까지만** 적용. 송금 단계엔 멱등 가드가 없었다.
- 배치에 실행 락(동시 실행 방지)이 없었다.

## 조치
- 즉시: 송금 직전 `payout_log`에 (가맹점ID, 정산일) UNIQUE 체크 추가. 이미 송금됐으면 스킵.
- 즉시: 배치에 분산 락(Redis) 도입 — 동시 실행 차단.
- 회수: 이중 송금분 3건 가맹점 협조로 당일 회수 완료.

## 교훈
- 멱등성은 "계산"이 아니라 **돈이 실제 나가는 지점**에 걸어야 한다. (INC-204와 동일 교훈 — 외부 부수효과 직전에 유니크 가드)
- 모든 배치는 단일 실행 락 필수.
