# 결제 백엔드 아키텍처 킥오프 회의

- 일시: 2026-03-04 14:00
- 참석: 김도현(테크리드), 박서연(백엔드), 이준호(백엔드), 최민지(PM)

## 안건
1. 결제 승인 플로우 동기/비동기 결정
2. PG사 연동 추상화 레벨
3. 멱등성 처리 방식

## 논의
- 김도현: PG 승인은 외부 지연이 크다(평균 800ms, p99 3s). 동기로 묶으면 우리 API가 PG에 인질로 잡힌다. 비동기 + 상태 폴링으로 가자.
- 박서연: 그러면 클라이언트가 결제 결과를 어떻게 받나? → 웹훅 + 클라 폴링 둘 다 제공하기로.
- 이준호: PG사 2곳(토스페이먼츠, 나이스페이) 동시 지원해야 함. adapter 패턴으로 PaymentGateway 인터페이스 추상화.
- 멱등성: 같은 주문에 결제가 두 번 들어오면 중복 결제 발생. 멱등성 키 필요. → 이건 다음 회의에서 상세 결정(action: 박서연 조사).

## 결정
- **결제 승인은 비동기 처리.** API는 즉시 `PENDING` 반환, 워커가 PG 승인 후 상태 갱신.
- **PG 연동은 PaymentGateway 인터페이스로 추상화.** 토스/나이스 각각 adapter 구현.
- 클라이언트 결과 수신: 웹훅(서버간) + 폴링(클라) 병행.
