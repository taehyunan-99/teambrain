# 결정: 결제 SDK v1 범위 확정

- 일시: 2025-12-03
- 결정자: 김도현

## 배경
가맹점이 우리 결제 API를 직접 붙이려면 결제 호출, 결과 폴링, 헤더 세팅을 다 직접 구현해야 한다. 연동 부담이 커서 도입 문턱이 높다. 클라이언트 SDK(JS)를 제공해 이 부담을 줄인다. 첫 버전이라 범위를 좁게 확정한다.

## 검토한 대안
1. **JS SDK만, 결제 호출/폴링 래핑 (채택)**: 결제 승인 요청과 결과 폴링을 한 번에 감싸고, 멱등 키를 SDK가 자동으로 생성해 헤더에 실어 보냄. 자동생성은 기본 on이되 가맹점이 직접 넘기면 그 키를 존중.
2. JS + 모바일(iOS/Android) 동시 출시: 인력 대비 범위 과다. v1에서는 탈락, 이후 검토.
3. 멱등 키 미생성 시 요청 거부(강제): SDK 우회·직접 호출하는 가맹점이 매번 키를 직접 만들어야 함 → 연동 마찰 크다. 도입 초기 거부감 우려로 탈락.

## 결정
- **v1 범위는 JS SDK로 한정.** 모바일은 별도 트랙.
- SDK는 **결제 호출 + 결과 폴링**을 래핑한다.
- **멱등 키 자동생성은 기본 on(권장), 강제는 아님.** 가맹점이 키를 직접 주면 그대로 사용, 없으면 SDK가 UUID를 만들어 채워 보냄.
- SDK를 안 쓰고 API를 직접 호출하는 경로는 막지 않는다(키 없이도 호출 가능).

## 근거
- 첫 SDK는 "쉽게 붙는다"가 최우선. JS만으로도 웹 가맹점 대부분을 커버.
- 폴링까지 감싸면 가맹점이 결과 확인 로직을 안 짜도 됨 → 연동 코드가 크게 준다.
- 키 자동생성을 기본 on으로 두면 SDK 사용자는 신경 안 써도 멱등성 이점을 얻음. 강제까지 하면 직접 호출하는 가맹점이 막혀 도입이 늦어진다.

## 리스크
- 가맹점이 SDK 없이 직접 호출하면서 키를 안 보내면 멱등성 보호를 못 받음 → 중복 결제 가능성 남음(권장이라 막지 않음).
- 자동생성 키를 가맹점이 요청마다 새로 만들어 끼면(예: 리트라이 루프 안에서 재생성) 멱등성이 무력화될 수 있음 → SDK 문서/가이드에 "재시도 시 같은 키 유지" 명시 필요.
- JS만 지원하므로 모바일 가맹점은 당분간 직접 연동해야 함.
