# 결정문서: 결제 API 컨테이너화 및 CI 파이프라인 도입

- 일시: 2024-09-16
- 결정자: 강민석 (플랫폼 엔지니어)
- 관련: 2024-09-10 수동 배포 사고 (#infra)

## 배경

- 2024-09-10 11시경 정유진이 결제 API 서버에 ssh로 접속해 수동 배포 진행.
- 의도와 다른 브랜치를 빌드/기동하여 결제 approve 엔드포인트가 약 5분간 응답 실패.
- 즉시 직전 정상 빌드로 롤백하여 복구. 가맹점 영향은 소규모였으나 결제 트래픽 구간에서 발생한 사고임.
- 원인: 배포가 사람 손에 의존(브랜치 선택, 빌드, 기동을 수기로 수행). 재현 가능한 빌드 산출물과 자동 검증 단계 부재.
- 현재 배포 방식: 서버에서 직접 git pull 후 systemd 서비스 재기동. 빌드 환경이 서버 상태에 의존하여 재현성이 낮음.

## 검토한 대안

### 대안 A. systemd + 배포스크립트 유지 (현행 보완)
- 내용: 현재 systemd 운영을 유지하고, 수동 ssh 대신 배포스크립트(브랜치 고정, 빌드, 헬스체크)로 휴먼에러를 줄인다.
- 장점: 추가 학습/인프라 비용 거의 없음. 즉시 적용 가능.
- 단점: 빌드 산출물이 여전히 서버 환경에 의존하여 재현성 미확보. "내 서버에선 됨" 문제 잔존. 자동 테스트 게이트가 파이프라인에 강제되지 않음. 사고의 근본 원인(사람 의존) 해소 한계.

### 대안 B. 바로 K8s 도입
- 내용: 컨테이너화와 동시에 오케스트레이션까지 K8s로 한 번에 전환.
- 장점: 오토스케일/롤링업데이트/셀프힐링 등 운영 이점을 한꺼번에 확보.
- 단점: 클러스터 구축/운영 학습 비용이 큼. 매니페스트/네트워킹/시크릿 관리 등 도입 표면적이 넓어 단기 안정성 리스크. 현재 팀 규모/숙련도 대비 과함.

### 대안 C. 단계적 — 컨테이너화 + CI 먼저, 오케스트레이션은 후행 (채택)
- 내용: 1단계로 결제 API를 Docker 이미지로 빌드하고 CI에서 빌드/테스트/이미지 푸시까지 자동화한다. 오케스트레이션(K8s)은 Q4에 별도로 진행한다.
- 장점: 재현 가능한 빌드 산출물 확보. 테스트 게이트를 파이프라인에 강제. 도입 표면적을 나눠 안정성 리스크 분산. 컨테이너 이미지는 K8s 전환의 사전 작업이 됨.
- 단점: 이미지 빌드/레지스트리 구성 등 초기 작업 필요. 1단계만으로는 배포 자동화/오토스케일은 미확보(수동 또는 스크립트로 이미지 기동).

## 결정

대안 C 채택. 단계적으로 진행한다.

1단계 (이번 도입):
- 결제 API를 Docker 이미지로 빌드한다 (Dockerfile 작성, 멀티스테이지 빌드로 런타임 이미지 경량화).
- CI 파이프라인에서 다음을 자동 수행한다.
  - build: 의존성 설치 및 빌드.
  - test: 단위/통합 테스트 실행. 실패 시 파이프라인 중단(머지 차단).
  - image push: main 머지 시 이미지 태그(커밋 SHA + 환경) 부여 후 레지스트리에 푸시.
- 배포는 당분간 푸시된 이미지를 기동하는 방식으로 전환한다. 서버에서 직접 git pull/빌드하던 흐름을 제거한다.

2단계 (Q4 예정, 별도 결정문서로 다룸):
- 컨테이너 기반이 안정화된 뒤 K8s로 오케스트레이션 전환을 진행한다.

## 근거

- 이번 사고의 근본 원인은 사람이 수기로 브랜치 선택/빌드/기동을 수행한 점이다. 빌드 산출물을 이미지로 고정하고 CI가 검증/푸시를 담당하면 휴먼에러 경로를 제거할 수 있다.
- K8s 직행은 운영 이점이 크지만 도입 표면적이 넓어 단기 안정성 리스크가 크다. 컨테이너화는 어차피 K8s의 선행 작업이므로, 먼저 이미지/CI를 안정화한 뒤 오케스트레이션으로 넘어가는 단계적 접근이 리스크 대비 효율이 높다.
- 안정성 우선 원칙에 따라, 한 번에 큰 변경을 넣기보다 검증된 단계를 쌓는다.

## 리스크 및 대응

- 이미지 빌드 시간 증가로 CI 파이프라인이 길어질 수 있음 → 멀티스테이지 빌드 및 레이어 캐시로 완화.
- 레지스트리/시크릿 관리 신규 도입 → 접근 권한 최소화, 자격증명은 CI 시크릿으로 관리.
- 1단계 단계에서는 이미지 기동이 여전히 수동/스크립트 의존 → 배포 절차를 문서화하고 기동 스크립트에 헬스체크 포함, K8s 전환 전까지의 임시 운영으로 한정.
- 컨테이너 환경과 기존 서버 환경의 동작 차이 가능성 → 스테이징에서 이미지로 충분히 검증 후 전환.
