# 결정문서 — 결제 API 오토스케일(HPA) 도입

- 일시: 2025-10-08
- 결정자: 강민석 (플랫폼 엔지니어)
- 관련팀: 인프라(#infra), 결제(#pay-dev)

## 배경

- 결제 API는 현재 고정 replica 수(3개)로 운영 중. 트래픽 변동에 수동 대응.
- 평소 대비 프로모션/이벤트 구간에서 요청량이 크게 튐. 사전 인지 못 한 급증 시 응답 지연·타임아웃 발생 우려.
- 수동 스케일은 야간·주말 대응 공백이 있고, SRE 온콜 부담이 큼.
- K8s 클러스터 위에서 운영 중이므로 표준 오토스케일 메커니즘 적용 가능.

## 검토한 대안

1. 수동 스케일 유지
   - 장점: 동작 예측 가능, 추가 설정 없음, 스케일 결정에 사람 판단 개입.
   - 단점: 급증 대응 지연(인지→투입까지 수분~수십분), 온콜 부담, 야간 공백.

2. 클러스터 오토스케일(Cluster Autoscaler)만 적용
   - 장점: 노드 부족 시 노드 자동 증설.
   - 단점: replica 수 자체는 늘지 않음. pod 레벨 수요 급증에 직접 대응 못 함. 노드 증설은 수 분 단위로 느림.

3. HPA(Horizontal Pod Autoscaler) 도입 — 채택
   - 장점: 부하 메트릭 기반으로 replica 수 자동 조정. 결제 API의 단기 트래픽 급증에 직접 대응.
   - 단점: 메트릭 임계치 튜닝 필요. pod만 늘려서는 해결 안 되는 병목(다운스트림) 존재 가능.

## 결정

결제 API에 HPA 도입.

- 타깃 메트릭: CPU 사용률 70% 타깃.
- 추가 메트릭: 커스텀 메트릭(RPS) 기반 스케일 검토. 메트릭 어댑터 구성·임계치 산정 후 단계 적용.
- replica 범위: min 2 / max 10.
- 클러스터 오토스케일과 병행 가능하나, 이번 결정 범위는 HPA(pod 레벨)로 한정.

## 근거

- 결제 API 부하는 CPU와 강한 상관. CPU 타깃이 1차 신호로 적합.
- 요청량 급증을 CPU가 후행 반영하는 경우가 있어, RPS 커스텀 메트릭을 보조 신호로 검토.
- min 2로 단일 pod 장애 시에도 가용성 확보. max 10은 현재 클러스터 노드 용량 기준 상한.

## 리스크

- pod를 늘려도 Redis/DB 커넥션이 병목이면 효과가 제한됨. replica 증가가 곧 처리량 증가로 이어진다는 보장이 없음. 다운스트림(Redis, MySQL) 커넥션 풀·용량을 같이 봐야 함.
- 커넥션 풀 상한을 넘는 pod 증설은 DB 커넥션 고갈로 오히려 장애를 키울 수 있음. max replica는 다운스트림 커넥션 한도와 연동해 산정 필요.
- CPU 타깃만 보면 I/O 대기 위주 부하에서 과소 스케일 가능 → RPS 메트릭 검토 사유.
- 스케일 인 시 진행 중 요청 유실 방지 위해 graceful shutdown·연결 드레이닝 점검 필요.
