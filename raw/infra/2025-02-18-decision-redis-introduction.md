# 결정: Redis 도입 — 캐시/세션 저장소 표준화

- 일시: 2025-02-18
- 결정자: 정유진(제안·운영), 강민석(클라이언트 연결 검토)

## 배경
서비스가 늘면서 캐시 계층과 세션 저장소가 팀마다 제각각이다. 일부는 애플리케이션 메모리에, 일부는 DB에 세션을 둔다. 인스턴스가 여러 개로 늘고 K8s 위에서 파드가 수시로 재기동되면서, 로컬 메모리 캐시·세션은 일관성이 깨진다. 공용 인메모리 저장소를 표준으로 정해 캐시와 세션을 한곳에서 다룰 필요가 있다.

## 검토한 대안
1. **Redis (채택)**: 관리형 Redis를 K8s 클러스터에 붙여 사용. 풍부한 자료구조(string/hash/sorted set 등), 키별 TTL 네이티브 지원, 운영 도구·관측성 성숙.
2. memcached: 단순 키-값 캐시로는 가볍고 빠름. 그러나 자료구조가 빈약하고 영속화·세션 활용에 제약. 세션까지 포괄하기엔 부족. 탈락.
3. DB 세션 테이블: 별도 인프라 없이 기존 MySQL에 세션 테이블. 그러나 세션 R/W가 DB 부하로 직결되고 만료 정리를 배치로 돌려야 함. TTL을 직접 구현해야 하는 운영 부담. 탈락.

## 결정
- **Redis를 캐시·세션 저장소의 사내 표준으로 도입.** 관리형 인스턴스를 K8s 워크로드에서 접근.
- 운영(인스턴스/알람/백업/배포)은 인프라팀(정유진). 클라이언트 연결·커넥션 풀 설정은 강민석.
- **단일 인스턴스로 시작.** 트래픽·가용성 요구가 커지면 추후 replica(읽기 분산·HA) 검토.
- **maxmemory는 일단 넉넉히** 잡아 초기 사용량 대비 여유를 둔다.
- **eviction 정책은 `noeviction`.** 메모리가 차도 키를 임의로 버리지 않고 쓰기를 거부한다. 데이터 유실 방지를 기본값으로 둔다.
- 키 네이밍은 `{도메인}:{용도}:{식별자}` 컨벤션, TTL은 용도별로 호출 측에서 명시.

## 근거
- 자료구조·TTL이 네이티브라 캐시와 세션을 한 저장소로 자연스럽게 수용.
- TTL 만료를 Redis가 관리하므로 별도 정리 배치가 필요 없음.
- 운영·모니터링 도구가 성숙해 SRE 입장에서 다루기 쉬움.

## 리스크
- 단일 인스턴스는 SPOF. 장애 시 캐시 미스·세션 끊김. → replica 도입 전까지는 캐시/세션 외 critical 경로 사용 자제.
- `noeviction` + 단일 인스턴스에서 메모리가 한계에 닿으면 쓰기 거부로 이어질 수 있음. → maxmemory 사용률 알람을 운영 초기부터 건다(action: 정유진).
- 커넥션 풀 설정 미흡 시 파드 증설에서 연결 수 폭증 가능. → 풀 상한·타임아웃 기본값 정리(action: 강민석).
