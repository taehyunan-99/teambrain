# 결정문서: MySQL 운영 표준화 — 백업/페일오버 체계

- 일시: 2025-05-20
- 결정자: 정유진 (인프라/SRE 리드)
- 관련팀: 결제팀, 정산팀, 인프라팀

## 배경

- 현재 MySQL은 K8s 클러스터 내부에 단일 인스턴스로 떠 있음. StatefulSet 1개, 백업은 수동 mysqldump 비정기 수행.
- 결제 승인/취소, 멱등성 키, 정산 payout 데이터가 모두 이 단일 노드에 집적되어 있음. 노드 장애 시 복구 절차 부재.
- 백업이 일 단위 정합성을 보장하지 못함. 특정 시점 복구(PITR) 불가 — 사고 발생 시 마지막 수동 덤프 시점까지만 되돌릴 수 있어 데이터 유실 구간이 큼.
- 결제/정산은 금액(DECIMAL) 데이터를 다루므로 정합성과 복구 가능성이 인프라 차원의 전제 조건임.
- 운영 책임 소재와 표준이 정의되어 있지 않아 장애 대응이 임기응변에 의존.

## 검토한 대안

### 대안 A — self-managed MySQL 복제 HA (K8s 내부)
- K8s 내부에 MySQL 비동기/반동기 복제로 primary/standby 직접 구성. PITR은 별도 binlog 아카이빙 스토리지 구축.
- 장점: 인프라 통제권 최대, 비용 절감 여지.
- 단점: 페일오버 자동화/binlog 아카이빙/백업 검증을 전부 인프라팀이 직접 운영. SRE 인력 부담 큼. 운영 미숙으로 인한 split-brain 등 위험을 자체 부담.

### 대안 B — 관리형 RDS급 (외부 관리형 서비스)
- K8s 외부의 관리형 MySQL로 분리. 자동 백업/PITR/standby 페일오버를 매니지드로 제공받음.
- 장점: 백업·페일오버·패치를 검증된 매니지드 운영에 위임. 인프라팀은 모니터링/연결 관리에 집중. SLA 확보.
- 단점: 비용 증가. 외부 종속성. 페일오버 시 엔드포인트 전환에 따른 짧은 커넥션 단절 발생.

## 결정

- 대안 B(관리형 RDS급)를 채택한다. MySQL을 K8s 외부 관리형 HA 구성으로 정식 분리·표준화한다.
- 구성 요소:
  1. 자동 일일 백업 + PITR(특정 시점 복구) 활성화. 백업 보존 기간 14일.
  2. primary/standby 다중 AZ 복제 구성, 자동 페일오버(standby 승격).
  3. 앱과 DB 사이에 ProxySQL 커넥션 풀 도입. 커넥션 수렴 및 페일오버 시 재연결 완충.
- 결제/정산 서비스의 DB 접속은 모두 ProxySQL 엔드포인트를 경유하도록 표준화한다.

## 근거

- 결제/정산 금액 데이터의 신뢰성 기반 확보가 최우선. PITR로 사고 시 데이터 유실 구간을 분 단위로 축소.
- 페일오버 자동화/백업 검증을 매니지드에 위임하여 제한된 SRE 인력을 모니터링·연결 관리에 집중.
- ProxySQL로 K8s 다수 파드의 커넥션 폭증을 제어하고, 페일오버 시 엔드포인트 전환 충격을 풀 계층에서 흡수.
- 비용 증가는 단일 노드 장애 시 결제 중단 리스크 대비 수용 가능 범위로 판단.

## 리스크

- 페일오버 발생 시 primary 엔드포인트 전환(standby 승격) 동안 짧은 커넥션 끊김(수 초~수십 초)이 불가피함. 이 구간의 인플라이트 쿼리는 실패할 수 있음.
  - 대응: 앱단에서 DB 커넥션 오류에 대한 재시도 로직 필요. 결제팀/정산팀은 트랜잭션 실패 시 멱등성 키 기반 안전 재시도가 가능한지 점검 요망.
- 비동기 복제 특성상 페일오버 시점에 standby로 아직 전파되지 않은 binlog 구간은 유실될 수 있음. 동기성 수준(반동기 복제 등) 선택은 별도 검토 필요.
- 외부 관리형 종속에 따른 비용/네트워크 레이턴시 증가. K8s ↔ DB 네트워크 경로 모니터링 추가.
- 백업/PITR 복구 절차는 정기적으로 복구 리허설을 수행해 실효성을 검증해야 함.
