# 결정문서 — 초기 인프라 베이스라인 원칙

- 일시: 2024-07-22
- 결정자: 정유진 (인프라/SRE 리드)
- 관련: #infra

## 배경

- 합류 첫 주 현황 파악 결과, 결제 API와 MySQL이 단일 EC2 인스턴스 1대에 함께 올라가 있음.
- 배포는 서버 SSH 접속 후 수동 빌드/재시작. 배포 중 결제 API 다운타임 발생.
- 모니터링/알람 없음. 서버 이상은 가맹점 문의나 직접 접속으로만 인지.
- DB 백업 스케줄 없음. 디스크 장애 시 결제/정산 데이터 전량 유실 가능.
- 단일 장애점(SPOF) 다수. 인스턴스 1대 다운 = 전체 서비스 중단.

## 검토한 대안

1. 즉시 컨테이너 + 멀티 인스턴스 전면 전환
   - 장점: SPOF 근본 해소.
   - 단점: 결제팀 인력 현재 결제 API 안정화에 집중. 인프라 전환에 동시 투입 불가. 일정 비현실적.

2. DB 즉시 분리(별도 인스턴스)
   - 장점: 데이터 격리, 리소스 경합 해소.
   - 단점: 마이그레이션 중 다운타임 + 롤백 리스크. 현 인력으로 안전 분리 어려움. 백업 없는 상태에서 분리 시도는 위험.

3. 단일 서버 유지 + 최소셋 안전장치부터 도입 (채택)
   - 장점: 즉시 적용 가능, 가장 큰 리스크(데이터 유실/무가시성)부터 제거.
   - 단점: SPOF는 당분간 잔존.

## 결정

1. 단일 서버 체제는 임시로 간주한다. Q4까지 컨테이너 기반(이미지 빌드 → 배포)으로 전환한다.
2. 모니터링/알람 최소셋을 먼저 도입한다. 1차 대상은 헬스체크(API 응답), 디스크 사용률, CPU 사용률 3종.
3. DB는 당장 물리 분리는 보류한다. 대신 MySQL 자동 백업 스케줄(일 1회 덤프 + 보관)을 즉시 건다.
4. 수동 SSH 배포는 Q4 컨테이너 전환과 함께 폐기 대상으로 둔다.

## 근거

- 데이터 유실은 복구 불가능한 사고다. 백업은 비용 대비 효과가 가장 큰 첫 조치다.
- 무가시성 상태에서는 장애를 사후에만 안다. 최소 알람 3종으로 능동 인지 전환.
- 컨테이너 전환은 SPOF/배포 다운타임을 동시에 푸는 가장 효율적인 경로지만, 인력 사정상 단계적으로 간다.
- 작게 시작해 검증된 것만 늘린다. 첫 단계에서 알람 항목을 욕심내지 않는다.

## 리스크

- 단일 서버 SPOF는 Q4 전환 전까지 잔존. 인스턴스 다운 시 전체 중단 가능 — 수용하고 진행.
- DB가 같은 인스턴스를 쓰는 한 결제 트래픽 급증 시 DB 리소스 경합 가능.
- 백업은 떠도 복구 절차를 검증하지 않으면 무의미. 백업 도입 후 복구 리허설을 별도 과제로 등록 필요.
- 알람 최소셋은 메모리 누수/커넥션 고갈 등 세밀한 이상은 못 잡음. 운영하며 항목 보강 예정.
