# [Decision] MySQL 프라이머리 페일오버 표준 정책

- 일시: 2025-05-19
- 결정자: 정유진(인프라/SRE 리드), 강민석(플랫폼 엔지니어)
- 관여팀: 인프라, 결제(#pay-dev), 정산(#settlement)
- 상태: 승인

## 배경

- 현재 MySQL은 프라이머리 단일 인스턴스 + 비동기 복제 스탠바이 1대 구성. 프라이머리 장애 시 복구 절차가 문서화되어 있지 않음.
- 결제/정산 모두 MySQL을 주 트랜잭션 저장소로 사용(금액 DECIMAL). 두 팀 모두 쓰기 경로가 핵심.
- 비동기 복제 특성상 프라이머리 장애 시점에 스탠바이로 아직 전파되지 않은 커밋이 유실될 수 있음. 결제 승인/정산 payout 같은 금전 트랜잭션에서는 허용 불가.
- 운영 중 수동 복구만 가능한 상태라 RTO가 길고, 야간/새벽 장애 시 대응 지연 우려.

## 검토한 대안

### 대안 A. 수동 페일오버 (운영자 판단 승격)
- 프라이머리 장애 감지 시 온콜이 수동으로 스탠바이를 승격.
- 장점: 승격 판단에 사람이 개입 → 스플릿브레인 위험 낮음, 오탐 페일오버 없음.
- 단점: RTO가 사람 대응 시간에 종속(새벽 10~30분+). 결제 가맹점 영향 시간 길어짐. 절차 휴먼에러 가능.

### 대안 B. 자동 페일오버 (헬스체크 기반 자동 승격)
- 헬스체크가 프라이머리 장애를 확정하면 자동으로 스탠바이 승격 + 라우팅 전환.
- 장점: RTO 단축(목표 수십 초~1분 내). 새벽 장애도 무인 대응.
- 단점: 헬스체크 오탐 시 불필요한 페일오버/스플릿브레인 위험. 펜싱(fencing)·쿼럼 설계 필요.

## 결정

**대안 B(자동 페일오버)를 채택**하되, 아래 제약을 표준으로 둔다.

1. **반동기 복제(semi-synchronous replication) 전환**
   - 프라이머리는 최소 1개 스탠바이의 binlog 수신 확인(ACK) 후 커밋 응답(`rpl_semi_sync_master_enabled=ON`, `rpl_semi_sync_master_wait_point=AFTER_SYNC`).
   - 스탠바이 ACK를 받은 트랜잭션은 페일오버 후에도 무손실에 수렴(RPO≈0). 결제 승인/정산 payout 등 금전 커밋의 유실을 원천 차단. GTID 기반 복제로 승격 후 위치 정합 보장.

2. **헬스체크 + 자동 승격 + 펜싱**
   - 헬스체크 다중 확인(연속 실패 임계치)으로 오탐 방지. 확정 시 스탠바이 자동 승격.
   - 구 프라이머리는 펜싱으로 강제 격리하여 스플릿브레인 방지. 쿼럼 기반 판정.

3. **페일오버 윈도우 동안 쓰기 거부**
   - 장애 감지~승격 완료 구간은 쓰기 라우팅을 차단(읽기 전용 또는 즉시 에러 반환).
   - 애플리케이션은 쓰기 실패를 "재시도 가능한 일시 오류"로 처리하고, 클라이언트에 성공을 절대 반환하지 않음.
   - 윈도우 목표: 수십 초~1분 이내.

## 근거

- 반동기 복제로 커밋 무손실에 수렴시키면, 자동 페일오버의 유일한 치명 리스크(커밋 유실)가 사실상 제거됨 → 자동화의 RTO 이점만 취할 수 있음.
- 결제/정산은 "성공 응답한 트랜잭션은 반드시 보존"이 회계 정합성의 전제. 순수 비동기 복제로는 이 전제를 깰 수 있어 수용 불가.
- 윈도우 동안 쓰기를 거부하는 편이, 불확실한 상태에서 쓰기를 받아 부분 커밋/모순을 만드는 것보다 안전(애매하면 막는다).

## 쓰기 경로 영향 (결제·정산 합의 사항)

- **결제 in-flight 트랜잭션**: 페일오버 윈도우에 진행 중이던 결제 승인 쓰기는 실패 처리됨. 승인 응답이 나가지 않은 건은 미승인으로 남으므로, 클라이언트 재시도 시 Idempotency-Key로 중복 방지 경로를 그대로 탄다. 쓰기 거부 구간에서 PG 어댑터/웹훅 처리는 일시 오류로 백오프 재시도.
- **정산 D+1 배치(04:00 KST)**: 배치 실행 중 페일오버가 발생하면, 진행 중 트랜잭션은 윈도우 동안 쓰기 거부로 실패. 배치 재개/재실행 시 이미 처리 완료된 정산을 다시 만들지 않도록 애플리케이션 레벨에서 처리 상태를 확인하고 멱등하게 재개해야 함(현재 DB 레벨 복합 유니크 가드는 없음 — 정산팀 별도 점검 필요).
- 두 팀 모두 "쓰기 실패 = 일시 오류로 간주, 절대 성공 처리 금지" 원칙에 합의.

## 리스크

- **반동기 복제 쓰기 지연**: 스탠바이 ACK 대기로 커밋 레이턴시 증가 가능. 특히 프로모션 등 트래픽 급증 시 쓰기 처리량 영향. → 스탠바이 네트워크/디스크 성능 확보, 모니터링으로 복제 지연 알람. ACK 타임아웃 시 비동기로 폴백되는 동작(`rpl_semi_sync_master_timeout`) 인지.
- **반동기 스탠바이 단독 장애 시 쓰기 지연/폴백**: ACK 대상이 1대뿐이면 그 스탠바이 장애 시 타임아웃까지 커밋이 지연되거나 비동기로 폴백됨. → ACK 대상 후보 다중화 검토.
- **헬스체크 오탐**: 네트워크 단절을 장애로 오인해 불필요 페일오버 가능. → 임계치/쿼럼/펜싱으로 완화, 페일오버 이벤트 알람 필수.
- **쓰기 거부 윈도우의 가맹점 체감 오류**: 짧지만 결제 실패가 표면화됨. → 윈도우 최소화 + 클라이언트 재시도 가이드.

## 후속 조치

- 인프라: 반동기 복제 구성 전환, 헬스체크/자동 승격/펜싱 파이프라인 구축, 페일오버 알람 정의(정유진/강민석).
- 결제: 쓰기 일시 오류에 대한 백오프 재시도 및 멱등 경로 점검(#pay-dev).
- 정산: 배치 중단/재개 시 멱등 재개 절차 확인, 페일오버 시 배치 재실행 절차 문서화(#settlement).
- 스테이징에서 페일오버 드릴(주입 장애) 후 RTO/RPO 측정 → 표준 수치 확정.
