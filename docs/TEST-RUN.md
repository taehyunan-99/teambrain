# wiki-build 증분 사이클 테스트 — 실행 가이드

> 다른 터미널 새 세션에서 따라 하는 절차. 목적: `wiki-build` 스킬이
> (1) 새 raw만 골라 증분 빌드하고 (2) 기존 아티클을 머지 갱신하며
> (3) 그 지식으로 질문에 답하는지 검증한다.

## 현재 상태 (이 세션에서 세팅 완료)

- 이 세션에서 **테스트용 raw 2개를 이미 생성**했다:
  - `raw/slack/2026-06-15-idem-ttl-48h-approved.md` — 기존 "TTL 48h 검토중"의 **결말(승인)**
  - `raw/2026-06-16-decision-multi-pg-routing.md` — **완전 새 주제**(멀티 PG 라우팅)
- vault는 **커밋 0개** 상태(전부 untracked).

## ⚠️ 먼저 이해할 것 — 왜 baseline 커밋이 필요한가

커밋이 0개면 `git status`에 **모든 파일이 `??`로** 뜬다 → 새 raw 2개만
골라낼 수가 없다. 그래서 **새 raw 2개를 뺀 나머지를 먼저 커밋**해야,
그 2개만 untracked로 남아 "새로 추가된 raw"로 깔끔하게 잡힌다.

---

## STEP 0 — 새 터미널에서 세션 시작

새 터미널을 열고:

터미널에 **이 두 줄을 직접** 입력한다(`.../`는 줄임이 아니라 실제 전체 경로다):

```
cd /Users/taehyunan/Desktop/Repo/Practice/wiki
claude
```

> 이 폴더에서 띄워야 `.claude/skills/wiki-build/`가 자동 인식된다.
> 매번 들어가기 귀찮으면(선택): `alias wiki='cd /Users/taehyunan/Desktop/Repo/Practice/wiki && claude'`

---

## STEP 1 — baseline 커밋 (새 raw 2개는 제외)

새 세션의 claude에게 **이대로** 입력:

```
지금부터 wiki-build 증분 테스트를 할 거야. baseline을 만들어줘.
단, 테스트용으로 새로 추가된 이 두 파일은 커밋에서 빼줘:
- raw/slack/2026-06-15-idem-ttl-48h-approved.md
- raw/2026-06-16-decision-multi-pg-routing.md
나머지 전부를 "baseline: 빌드 완료 상태" 메시지로 커밋해줘.
커밋 후 git status로 그 두 파일만 untracked로 남았는지 보여줘.
```

**✅ 확인할 것:**
- 커밋이 1개 생성됨
- `git status`에 **딱 그 2개 파일만** `??`(untracked)로 남음

---

## STEP 2 — wiki 빌드 실행

claude에게:

```
wiki 빌드해줘
```

> "wiki 빌드 / 위키 업데이트 / build wiki / wiki 갱신" 아무거나 트리거됨.

**✅ 확인할 것 (멱등성 + 증분):**
- 새 raw **2개만 dirty**로 잡고, 기존 raw 다수는 hash 일치로 **스킵**한다고 보고하는가
- 기존 아티클 중 **건드리지 않은 것들**을 스킵으로 보고하는가
- `idempotency.md`는 **머지 갱신**(TTL 항목), 멀티 PG는 **신규 생성**으로 가는가
- (커밋은 스킬이 안 한다 — working tree만 바뀜)

---

## STEP 3 — 빌드 결과 점검

claude에게:

```
이번 빌드에서 뭐가 새로 생기고 뭐가 갱신됐는지 git status랑 git diff로 보여줘.
그리고 dangling wikilink(존재하지 않는 개념을 가리키는 [[링크]])가 없는지 확인해줘.
```

**✅ 확인할 것:**
- `idempotency.md`의 TTL 서술이 "48h 검토중" → **"승인/적용"**으로 바뀌었는가
  (redis만 48h, DB 유니크는 영구 유지라는 내용까지 반영되면 더 좋음)
- 멀티 PG 라우팅 아티클(예: `multi-pg-routing.md`)이 **새로 생기고**,
  `payment-gateway-abstraction`과 **상호 링크**됐는가
- `index.md` 카탈로그에 새 개념이 추가됐는가
- dangling wikilink **0개**

---

## STEP 4 — 질문 (핵심 검증, raw 차단)

이제 빌드된 wiki만으로 답하는지 본다. **공정성을 위해 raw를 못 읽게 막는다.**
claude에게 **이 블록 전체**를 입력:

```
지금부터 너는 wiki/ 폴더와 index.md 만 읽을 수 있어. raw/ 는 절대 읽지 마.
아래 질문에 "해결방향 / 근거(과거 사건·결정) / 출처(어느 아티클의 어느 raw로
역추적되는지)" 형식으로 답해줘. wiki에 정보가 없으면 "정보 없음"이라고 해.

질문:
1. redis가 느려지면서 중복 결제가 의심돼요. 전에 이런 적 있었나요? 어떻게 대응하죠?
2. 새벽에 정산 배치가 40분째 안 끝나요. 죽은 것 같은데 재실행해도 되나요?
3. 멱등성 키 TTL은 지금 몇 시간이에요? 바뀐 적 있어요?
4. 멀티 PG 라우팅 같은 거 계획 있어요? 언제, 왜 하나요?
```

**✅ 채점 기준:**
| 질문 | 합격 신호 |
|------|-----------|
| Q1 | 기존 지식 회수 — INC-204(중복결제 87건) / fail-closed / DB 유니크 최종방어선 |
| Q2 | 기존 지식 회수 — INC-231 "재실행 금지" 류의 정산 배치 가드 |
| **Q3 (결정타)** | **새 raw 반영** — "원래 24h였는데 48h로 승인됨(redis만, +12% 메모리)" **히스토리**까지 답하면 증분 빌드가 지식에 녹은 것 |
| Q4 | **신규 아티클에서 답** — Q3 착수, 토스 수수료 인상 대비, 어댑터 위 라우터 |

---

## STEP 5 — 정리 (선택)

테스트 결과를 남기거나 되돌린다. claude에게:

```
# (A) 결과를 남기려면
이번 테스트 변경 전부 git에 커밋해줘. 메시지는 "test: 증분 빌드 사이클 검증"

# (B) baseline으로 되돌리려면
git diff로 변경 확인하고, STEP 1의 baseline 커밋 상태로 되돌려줘
```

---

## 합격 기준 요약

5개 중 **4개 이상** 충족이면 "데이터 추가 → 증분 빌드 → 머지/신규 → 질문 답변"
전체 사이클이 동작하는 것이다.

| 단계 | 합격 신호 |
|------|-----------|
| STEP 2 | 새 raw 2개만 dirty, 나머지 스킵 (멱등성) |
| STEP 2 | 기존 아티클 미변경 + idempotency 머지 + multi-pg 신규 |
| STEP 3 | dangling 0, index.md에 새 개념 반영, 상호 링크 생성 |
| STEP 4 Q3 | TTL 변경 **히스토리**(24h→48h 승인)까지 답함 |
| STEP 4 Q4 | 새로 만든 아티클에서 답을 찾음 |

## 주의 (꼭 읽기)

- **이번(현재) 세션의 답변은 신뢰하지 말 것.** 이 세션은 raw를 직접 만들어서
  wiki 없이도 답을 안다. 새 세션은 그 기억이 없으니 **진짜 wiki만으로 답하는지**를
  공정하게 본다 — 그게 STEP 4의 가치다.
- 버그가 나오면 그 버그만 기록해 와라. "테스트 → 버그 수정 → 재테스트"가 다음 사이클.
