# wiki-build 전체 사이클 테스트 시나리오

> 새 터미널 세션에서 **데이터 추가 → 빌드 → 질문 → 답변**까지 한 바퀴 도는 절차.
> 목적: `wiki-build` 스킬이 (1) 새 raw를 감지해 (2) 개념 아티클로 합성하고 (3) 그 지식으로 질문에 답하는지 검증.

---

## 사전 이해 (왜 새 raw가 필요한가)

현재 vault는 **이미 빌드된 상태**(아티클 10개 / raw 86개, source_hash 일치).
그냥 빌드하면 멱등성 때문에 "변경 없음, 스킵"만 나온다.
→ 전체 사이클을 보려면 **새 raw를 1개 추가**해 증분 빌드를 유발해야 한다.

---

## STEP 0 — 새 세션 시작

새 터미널을 열고 vault 폴더에서 claude를 실행한다.

```
cd /Users/taehyunan/Desktop/Repo/Practice/wiki
claude
```

> 이 폴더에서 띄워야 스킬(`.claude/skills/wiki-build/`)이 자동 인식된다.

(선택) 깨끗한 기준선이 필요하면, 테스트 전에 현재 상태를 한 번 커밋해두면
나중에 `git diff`로 "이번 테스트가 만든 변경"만 깔끔히 볼 수 있다. 새 세션에서:

```
이 vault의 현재 상태를 git에 커밋해줘. 메시지는 "baseline: 비정형 빌드 완료"
```

---

## STEP 1 — 새 raw 데이터 추가 (사람이 원자재 투입)

claude에게 자연어로 새 노트를 만들게 하거나, 직접 파일을 둔다.
**테스트 포인트를 심으려면** 기존 개념과 엮이는 내용 + 새 사실을 섞는 게 좋다.

예시 — claude 프롬프트:

```
raw/slack/2026-06-15-inc204-retro-followup.md 파일을 만들어줘. 내용은
멱등성 TTL 48시간 연장이 드디어 승인된 슬랙 스레드야. redis 메모리 추산
결과 +12% 라서 감당 가능하고, 다음 배포에 포함하기로 결정. 형식은 기존
raw/slack/*.md 들과 똑같이 frontmatter(created/tags/channel) + 대화체로.
```

> 이 예시는 의도적으로 **"TTL 48h 검토중"이라는 기존 미결 사안의 결말**을 담는다.
> 빌드 후 `idempotency.md`의 미결 항목이 "승인됨"으로 갱신되는지가 관전 포인트.

추가로 1개 더(완전히 새로운 주제)를 넣으면 "신규 아티클 생성"도 같이 테스트된다:

```
raw/2026-06-16-decision-multi-pg-routing.md 도 만들어줘. 멀티 PG 라우팅을
Q3에 시작하기로 한 결정 문서야. 토스 수수료 인상 대비, 결제별로 토스/나이스를
고르는 라우팅 레이어를 어댑터 추상화 위에 얹는다는 내용. 형식은
raw/*-decision-*.md 들과 동일하게.
```

---

## STEP 2 — wiki 빌드 실행 (스킬 트리거)

claude에게:

```
wiki 빌드해줘
```

> "wiki 빌드 / 위키 업데이트 / build wiki / 위키 갱신" 중 아무거나 트리거된다.

**기대 동작 (7단계):**
1. 변경 감지 → 새 raw 1~2개만 dirty로 잡고 나머지 84개는 hash 일치로 스킵
2. 새 raw의 소스 요약 생성 (`wiki/sources/...`)
3. 개념 합성 — `idempotency.md` 머지 갱신(TTL 미결→승인), `multi-pg-routing.md` 신규 생성
4. wikilink 연결
5. index.md 갱신
6. frontmatter 위생(updated/source_hashes)
7. `_log.md` 기록 + 요약 보고 (커밋은 안 함)

**확인할 것:**
- "변경 없음 스킵 84개 / 신규·갱신 N개" 식으로 보고하는가 (멱등성 동작)
- 기존 9개 아티클을 건드리지 않는가 (불필요한 재작성 없음)

---

## STEP 3 — 빌드 결과 점검

claude에게:

```
이번 빌드에서 뭐가 새로 생기고 뭐가 갱신됐는지 git diff로 보여줘.
그리고 dangling wikilink 없는지 확인해줘.
```

**확인할 것:**
- `idempotency.md`의 TTL 항목이 "48h 검토중" → "승인/적용 예정"으로 바뀌었는지
- `multi-pg-routing.md`가 새로 생기고 `payment-gateway-abstraction`과 상호 링크됐는지
- index.md 카탈로그에 새 개념이 추가됐는지

---

## STEP 4 — 질문 (신입 시뮬레이션, 핵심 검증)

이제 빌드된 wiki만으로 답하는지 본다. **공정한 테스트를 위해** raw를 직접 읽지 말고
wiki만 보고 답하라고 제약을 건다.

```
지금부터 너는 wiki/ 폴더와 index.md 만 읽을 수 있어. raw/ 는 절대 읽지 마.
아래 질문에 "해결방향 / 근거(과거 사건·결정) / 출처(어느 아티클의 어느 raw로
역추적되는지)" 형식으로 답해줘. wiki에 없으면 "정보 없음"이라고 해.

질문:
1. redis가 느려지면서 중복 결제가 의심돼요. 전에 이런 적 있었나요? 어떻게 대응하죠?
2. 새벽에 정산 배치가 40분째 안 끝나요. 죽은 것 같은데 재실행해도 되나요?
3. 멱등성 키 TTL은 지금 몇 시간이에요? 바뀐 적 있어요?
4. 멀티 PG 라우팅 같은 거 계획 있어요?
```

**채점 기준:**
- Q1·Q2: 기존 빌드에 있던 지식(INC-204 fail-closed / INC-231 "재실행 금지") 회수
- Q3: **이번에 추가한 새 raw가 반영됐는지** — "원래 24h였는데 48h로 승인됨" 식으로 히스토리까지 답하면 증분 빌드가 지식에 녹은 것
- Q4: STEP 1에서 넣은 multi-pg-routing 신규 아티클을 찾아 답하는지

---

## STEP 5 — 정리 (선택)

테스트가 끝나면 변경을 커밋하거나, 되돌린다.

```
# 테스트 결과를 남기려면
이번 테스트 변경 전부 git에 커밋해줘. 메시지는 "test: 증분 빌드 사이클 검증"

# 테스트 전으로 되돌리려면 (baseline 커밋이 있을 때)
git diff 로 변경 확인 후, STEP 0의 baseline 커밋 상태로 되돌려줘
```

## 합격 기준 요약

| 단계 | 합격 신호 |
|------|-----------|
| STEP 2 | 새 raw만 dirty, 나머지 84개 스킵 (멱등성) |
| STEP 2 | 기존 9개 아티클 미변경, 1개 머지 + 1개 신규 |
| STEP 3 | dangling 0, index.md에 새 개념 반영 |
| STEP 4 Q3 | TTL 변경 **히스토리**(24h→48h 승인)까지 답함 |
| STEP 4 Q4 | 새로 만든 아티클에서 답을 찾음 |

5개 중 4개 이상이면 "데이터 추가 → 빌드 → 질문 → 답변" 전체 사이클이 동작하는 것.
